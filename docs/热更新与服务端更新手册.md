# 热更新与服务端更新手册

> 本文档写给**主库管理员（昴君）**与协作的 AI：当改了代码、想让手机/电脑端用上新功能时，**该做什么、不该做什么**。
>
> 一句话结论：**手机端 JS 改动走 EAS 热更新；服务端改动只需在本机重新构建并重启——两者都不需要碰云服务器。**

---

## 一、架构原理（为什么不用动云端）

```
手机 App ──┐
           ├──► 云服务器 8.166.122.38:9091（只做转发/反代，不跑业务）──► 本机 9091（真正的业务服务端 node dist/index.js）──► SQLite
电脑桌面端 ─┘
```

- **云服务器只是"传话筒"**：负责把公网请求转发到本机，本身不运行业务逻辑、不存数据。
- **真正的业务服务端跑在本机**：`server/dist/index.js`，监听 `0.0.0.0:9091`，数据库 `server/data/`。
- 因此：
  - 改**服务端代码** → 只需**本机重新 build + 重启本机进程**，云端无需任何操作。
  - 改**手机端 JS** → 走 **EAS 热更新**下发，连本机服务端都不用动。
  - `server/DEPLOY.md` 是「如何把 server 整体搬上云」的方案文档，**当前并未采用**（当前是云端转发 + 本机业务）。

> ⚠️ 易踩的坑：看到 `DEPLOY.md` 容易误以为业务在云端、要 `git pull` 部署。**不是的。** 业务在本机。

## 二、手机端热更新（改了 client/ 的 JS/TS）

适用：界面、`client/utils/` 逻辑等**纯 JS/TS 改动**（不含新增原生模块 / 改 app.config 插件 / 升级 SDK）。

### 命令

```bash
cd client
npx eas-cli update --branch preview --message "feat: 改动简述" --non-interactive
```

### 关键参数（已固定）

| 项 | 值 |
|---|---|
| channel / branch | `preview`（目前唯一） |
| runtimeVersion | `1.0.0` |
| EAS projectId | `8ddfb094-5c7b-4bf2-8faf-a841593477a7` |
| 账号 owner | `aers`（aershu@outlook.com） |
| 配置文件 | `client/app.config.ts`（不是 app.json）：`updates.url` + `checkAutomatically: ON_LOAD` |

### 手机端拉取方式

1. **彻底退出 App**（从后台任务列表划掉，不是只按 Home）
2. **冷启动** App → 此时在后台下载更新
3. 若没立刻变化，**再退出重启一次**（`ON_LOAD` 是"本次启动下载、下次启动应用"）

### 什么时候热更新**不够**、必须重新打包 APK

- 新增/升级了原生依赖（`npx expo install` 装的带原生代码的库）
- 改了 `app.config.ts` 里的插件、权限、包名、runtimeVersion
- 升级了 Expo SDK
> 这些情况要 `eas build`，不能只 `eas update`。

## 三、服务端更新（改了 server/ 的代码）

适用：`server/src/**` 的任何改动（路由、仓储、数据库结构等）。

### 步骤

```bash
# 1. 本机重新构建（node 进程不会自动重读 dist，必须 build）
cd server
pnpm run build          # 看到 "⚡ Build complete!" 即成功

# 2. 重启本机服务端进程
#    本机有守护机制：kill 掉旧进程后会自动拉起新的 node dist/index.js
#    先找到监听 9091 的进程 PID：
netstat -ano | grep ':9091' | grep -i listen
#    然后停掉它（PowerShell）：
powershell -Command "Stop-Process -Id <PID> -Force"
#    守护机制会自动拉起新进程；若没有，则手动后台启动：
#    node dist/index.js   （会自动加载 server/.env 里的 PORT / TOKEN_SECRET）
```

### 验证新代码已生效（用事实，不靠猜）

```bash
# 新增的路由：返回 401(需鉴权)=路由存在=新代码已上线；返回 404=还是旧代码
curl -s -o /dev/null -w "%{http_code}\n" -X POST \
  http://localhost:9091/api/v1/buildings/test/floors/insert \
  -H "Content-Type: application/json" -d '{}'
```

> `server/data/` 数据库不在 git 里，重新 build / 重启**不会动租客数据**，放心操作。

## 四、本次（长按楼层增删层）的完整复盘

**做了什么**：手机端长按楼层名 → 弹菜单（向上加层 / 向下加层 / 删除整层，删除带二次确认）。改动涉及：
- 客户端：`client/screens/building/index.tsx`、`client/utils/roomTypes.ts`、`client/utils/storage.ts`
- 服务端：`server/src/db/repo.ts`、`server/src/routes/data.ts`（新增 `POST /buildings/:id/floors/insert`、`DELETE /buildings/:id/floors/:floor`）

**正确的发布动作**：
1. 手机端：`eas-cli update --branch preview`（已做）
2. 服务端：本机 `pnpm run build` + 重启 9091 进程（已做）
3. 验证：新路由 curl 返回 401 ✓

**踩过的坑（教训）**：
- ❌ 误以为业务在云端 → 想去 `git push` + 云端部署。**错。业务在本机，云端只转发。**
- ❌ 改完只跑了 `tsc --noEmit` 没 `build` → dist 还是旧的。**改完必须 build。**
- ❌ node 进程启动后不重读 dist → **必须重启进程**才生效。
