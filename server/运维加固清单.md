# 上线前运维加固清单（待处理）

> 整理时间：2026-06-24
> 背景：通讯功能（申请查看 + 账号级授权）已开发并本地测试通过。
> 在把 server 部署到公网云服务器**之前**，建议按本清单加固。
> 当前规模（一个房东 + 几个账号、几十栋楼、低并发）下，正常使用**不会崩溃**；
> 本清单针对的是"运维事故"和"安全风险"，不是性能问题。

---

## 总体结论

按实际使用规模，SQLite + 单进程 Node 架构**完全够用，正常使用不会崩**。
以下不是"架构有问题"，而是"上公网前该补的运维/安全短板"。

### 这些在你的规模下**不用担心**（别过度设计）
- **SQLite 单文件**：WAL 模式下并发读无压力，写串行，几十万行都没问题。
- **同步数据库查询**：只有每秒几百请求才会阻塞，个位数并发无感。
- **客户端逐栋拉楼房（N+1 请求）**：几十栋眨眼返回，上百栋才需优化。

## 待处理项（按优先级）

### ① 自动备份 🔴 高优先级
- **风险**：服务器硬盘损坏 / 误删 = 租客数据全部丢失，不可恢复。
- **现状**：`DEPLOY.md` 第八节有备份脚本说明，但需要真正配置上 cron 定时任务。
- **建议做法**：
  - 服务器上 crontab 每天定时 `cp housing.db` 到 backups 目录。
  - 定期 `scp` 下载一份到本地电脑（异地备份）。
  - 可考虑保留最近 N 天的滚动备份。
- **工作量**：小（运维配置，无需改代码）。

### ② 强制设置 TOKEN_SECRET 🔴 高优先级
- **风险**：若用默认密钥，攻击者可伪造登录 token，冒充任意账号查看/篡改所有数据。
- **现状**：`crypto.ts` 里没设 `TOKEN_SECRET` 时仅打印警告，仍会用不安全默认值启动。
- **建议做法**：生产环境（NODE_ENV=production）下若未设 `TOKEN_SECRET`，**直接拒绝启动**并报错提示。
- **工作量**：小（server 启动时加一个检查）。

### ③ 崩溃兜底 / 全局错误处理 🟠 中优先级
- **风险**：
  - 某个未预料的错误可能让整个 Node 进程挂掉（虽有 pm2 自动重启，但会有短暂中断）。
  - 错误堆栈可能直接返回给前端，泄露内部信息。
- **现状**：没有 Express 全局错误中间件，也没有 `process.on('uncaughtException'/'unhandledRejection')` 兜底。
- **建议做法**：
  - 加 Express 全局错误处理中间件，统一返回简洁错误信息，不泄露堆栈。
  - 加进程级 `uncaughtException` / `unhandledRejection` 捕获，记录日志但不让进程异常退出。
- **工作量**：小。

### ④ 登录接口限流 🟠 中优先级
- **风险**：
  - 登录/注册接口无限流 = 可被暴力破解密码。
  - 密码哈希（scrypt）是 CPU 密集操作，疯狂调用登录还能拖慢/拖垮服务器（DoS）。
- **现状**：无任何限流。
- **建议做法**：对 `/auth/login`、`/auth/register` 加简单限流（如同一 IP 每分钟最多 N 次）。
- **工作量**：中（引入轻量限流中间件，注意 express 4 的兼容版本）。

### ⑤ node:sqlite 是实验特性 🟡 低优先级
- **风险**：`node:sqlite` 在 Node 24 仍是实验特性，未来升级 Node 大版本可能 API 变动导致跑不起来。
- **现状**：依赖 Node 24 自带的 `node:sqlite`。
- **建议做法**：
  - **部署时锁定 Node 版本**（如固定 Node 24.x），不随意升级大版本。
  - 升级 Node 前先在测试环境验证。
  - 写进部署文档作为运维纪律。
- **工作量**：无需改代码，属运维纪律。

## 其他可选增强（非必须，规模变大再考虑）

- **请求体大小限制**：当前 `express.json({ limit: '50mb' })` 偏大，可调小防止内存耗尽攻击。
- **结构化日志**：目前是 console 打印，量大后可换成带轮转的日志方案，便于排查。
- **健康检查 + 监控告警**：服务挂了能收到通知（如 uptime 监控）。
- **优雅关闭**：进程退出前做 SQLite WAL checkpoint，确保数据落盘。

## 实施建议

- **最低限度（强烈建议）**：① 自动备份 + ② 强制 TOKEN_SECRET —— 挡住"数据全丢"和"账号被冒充"两个最致命事故。
- **推荐**：再加 ③ 崩溃兜底 + ④ 登录限流。
- 以上均为**纯 server 端加固**，不动已测试通过的业务逻辑，不影响现有功能。
- ⑤ 不改代码，部署时锁 Node 版本即可。
